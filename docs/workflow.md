# 자가 개선 루프 워크플로

> 이 프로젝트는 [Claude Code 하네스 엔지니어링 플랜](~/.claude/plans/claude code 하네스 엔지니어링.md)의
> 자가 개선 루프를 적용한다. 매 작업마다 평가를 누적하고, 누적된 데이터를
> Opus가 회고해서 [claude.md](./claude.md)를 다듬는 구조.
>
> 이 파일은 **하네스 운영 매뉴얼**이다. 작업 흐름, 슬래시 커맨드 사용 시점,
> 회고 결과를 어떻게 반영하는지 정리한다.

---

## 0. 한 줄 요약

```
작업 → /eval → ccusage 차분 → evaluations.md 누적 → /eval-review (1~2주) → claude.md 수정 → 반복
```

## 2. 슬래시 커맨드 셋업

### 2.1 `/eval` — 작업 평가 기록

**언제**: 매 작업 (마일스톤 또는 단계) 완료 직후.

**무엇을 함**: 양식대로 평가하고 `~/.claude/evaluations.md`에 append.

**설치 (필요 시)**: `~/.claude/commands/eval.md` 에 다음 내용 저장.

````markdown
---
description: 작업 평가 + 로그 파일에 저장
---

방금 수행한 작업을 아래 양식으로 평가하고,
평가 결과를 `~/.claude/evaluations.md` 파일에 append 해줘.

파일이 없으면 생성. 각 평가 위에 헤더 `## YYYY-MM-DD HH:MM` 추가.
기존 내용은 건드리지 말고 파일 맨 끝에 추가만.

---

**작업 요약**: (한 줄)

**난이도**: 상 / 중 / 하
- 상: 시행착오 3회+, 구조 재설계 필요했음
- 중: 시행착오 1-2회, 대체로 계획대로
- 하: 한 번에 완료, 단순 편집

**사용 모델**: (Sonnet / Opus / Haiku 조합)

**불확실 지점**:
- (검증 못 한 것, 확신 없는 부분. 없으면 "없음")

**모델 적정성**: 적정 / 오버 / 언더
- 언더라면 어느 지점에서 상위 모델이 나았을지

**반복 패턴 후보**:
- (이 작업과 비슷한 성격의 작업에서 공통으로 통할 만한 규칙 1개, 없으면 "없음")

---

원칙:
- "모두 잘됐습니다" 류 금지
- 불확실 지점은 반드시 한 번 검토
- 솔직하되 과장된 자기비판도 금지 (진짜 문제 없으면 "없음")
- 평가 출력 후 반드시 파일에 append 수행 (Bash 또는 Write tool 사용)
````

### 2.2 `/eval-review` — Opus 회고

**언제**:
- 평가 20건 이상 누적
- 또는 1~2주 간격 정기 실행
- 또는 "뭔가 자꾸 같은 실수 반복되네" 감이 올 때

**왜 Opus**: 단건 평가에 안 보이는 패턴 추출 + 메타 분석 + 트레이드오프 판단.
Sonnet은 표면적 분석에 그침.

**설치 (필요 시)**: `~/.claude/commands/eval-review.md` 에 다음 내용 저장.

````markdown
---
description: 누적된 평가 로그를 Opus가 회고하고 개선안 제안
---

다음을 Opus 모델로 수행:

`~/.claude/evaluations.md` 파일을 읽고 다음 분석을 수행한다.

## 1단계: 파일 확인
- 평가 건수, 기간, 난이도 분포

## 2단계: 메타 집계
- 총 평가 건수, 기간
- 난이도 분포 (상/중/하 비율)
- 모델 적정성 분포 (적정/오버/언더)
- 사용 모델 조합별 빈도

## 3단계: 패턴 분석

A. **불확실 지점 TOP 3**
- 가장 자주 나타나는 미검증 영역
- 각 영역의 발생 횟수와 대표 예시

B. **언더스펙·오버스펙 발생 패턴**
- 모델 선택이 잘못된 케이스
- 공통 원인

C. **난이도 "상" 공통점**
- 어려운 작업의 구조적 원인
- claude.md 규칙이 있었는지, 적용됐는지

D. **반복 패턴 후보 누적**
- 4건 이상 반복된 패턴
- 각 패턴의 구조적 원인

## 4단계: 하네스 개선 제안

- claude.md 추가/수정 제안 (근거 명시)
- critical-notes.md 추가 제안 (실제 사고 기반만)
- 슬래시 커맨드 추가 제안
- "관찰만" 항목 (1~2건짜리 패턴, 더 데이터 필요)

## 5단계: 3줄 요약
- 가장 두드러진 패턴
- 가장 중요한 개선 액션
- 다음 회고까지 관찰 포인트

---

원칙:
- 근거 없는 제안 금지. 몇 건 데이터 기반인지 명시.
- 1~2건짜리 패턴은 "관찰만"으로 분류하고 즉시 반영하지 말 것.
- "오버" 케이스가 적다고 settings 조정 권하지 말 것.
- 사용자가 직접 수정할 수 있게 구체적 문구·위치까지 제안.
````

## 3. `ccusage` — 비용·토큰 추적

### 3.1 무엇인가
Claude Code 사용량 추적 외부 도구. 모델별 토큰·비용을 일자별·세션별로 표시.

### 3.2 왜 필요한가
같은 작업이라도 컨텍스트 누적, 모델 선택, thinking 토큰에 따라 비용 크게 변동.
추적 안 하면 "왜 오늘 비싸지?" 원인 추적 불가.

### 3.3 활용

**작업 단위 추적**
- 작업 시작 시점 비용 기록
- 작업 끝 시점 비용 확인
- 차분으로 그 작업의 비용 산출

**모델 분포 확인**
- Sonnet/Opus/Haiku 비중
- "이 작업은 왜 Sonnet이 95%였지?" → 패턴 발견

## 4. 회고 결과 → 문서 반영 규칙

`/eval-review` 결과를 받으면 다음 순서로 처리.

### 4.1 4건 이상 반복 패턴만 채택

[claude.md](./claude.md) §5에 명시. 1~2건짜리는 [process.md](./process.md) §회고 누적의
"관찰만" 칸에 메모하고 다음 회고에서 재확인.

### 4.2 어느 파일에 반영하는가

| 회고 결과 유형 | 반영 파일 |
|---|---|
| 운영 규칙, 모델 선택, 컨벤션 | [claude.md](./claude.md) |
| 실제 발견된 사고/버그 패턴 | [critical-notes.md](./critical-notes.md) |
| 다음 작업 진입점·체크포인트 | [process.md](./process.md) |
| 슬래시 커맨드 추가/수정 | `~/.claude/commands/` + 이 파일 §2 |
| settings.json 조정 | 사용자에게 제안만 (자동 변경 금지) |

### 4.3 claude.md 100줄 유지

새 규칙 추가 시 효과 없는 기존 규칙 1개 제거 검토. 총량 유지.

### 4.4 회고 결과 요약은 process.md §8에 1행

상세는 `~/.claude/evaluations.md`. process.md에는 1행만:
회고 일자 / 누적 건수 / 채택된 패턴 / 반영 파일.

## 5. 운영 원칙 (요약)

이 5가지는 [Claude Code 하네스 엔지니어링 플랜](~/.claude/plans/claude code 하네스 엔지니어링.md) §운영 원칙의 복사.

1. **데이터 없는 규칙은 추가하지 않는다** — 4건 이상 반복 패턴만 채택.
2. **claude.md는 80~120줄 유지** — 너무 길면 모든 규칙이 희석.
3. **자동 트리거에 의존하지 않는다** — 명시 호출이 신뢰성 높다.
4. **슬래시 커맨드 적극 활용** — 작업 유형별로 분리.
5. **eval 양식의 솔직함이 핵심** — 불확실/실패를 정직히 기록해야 회고 가치.
